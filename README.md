`DSCLUBOK` — evidence-first платформа для работы с научно-техническим корпусом.
Проект помогает инженерным и исследовательским командам задавать вопросы на естественном языке, получать ответы с опорой на документы, видеть карту связей, сравнивать подходы, находить пробелы и курировать факты в одном рабочем пространстве.

## Что это за продукт

`Научный клубок` — не просто чат над документами и не административная панель.
Это R&D copilot для сложных инженерных сценариев, где важно не только получить ответ, но и понять:

- на каких источниках он основан;
- какие факты и параметры были извлечены;
- есть ли противоречия;
- как связаны материалы, процессы, условия и результаты;
- что еще нужно проверить до принятия инженерного решения.

## Почему проект сильный

- `Evidence-first подход`
  Ответ строится не из "догадок интерфейса", а из найденных фрагментов, извлеченных фактов и связанных документов.
- `Граф знаний как рабочий инструмент`
  Пользователь видит не просто список файлов, а карту связей между материалами, процессами, публикациями, параметрами и условиями.
- `Source ingestion как first-class capability`
  Система умеет не только отвечать по текущему корпусу, но и добавлять новые документы через UI, folder import и curated manifests.
- `Устойчивость для демо и пилота`
  Есть curated deep-ingest сценарий, degraded mode, production compose и явная стратегия запуска на одном сервере.
- `Расширяемая архитектура`
  Новые парсеры, типы источников, внешние коннекторы и answer-моды можно добавлять без переписывания ядра.

## Что умеет платформа

- задавать инженерные вопросы на русском и английском языках;
- получать краткий ответ с доказательной базой;
- открывать подтверждающие фрагменты и источники;
- строить knowledge graph по текущему запросу;
- сравнивать технологии, документы и подходы;
- показывать contradictions и research gaps;
- искать экспертов и релевантные направления;
- добавлять новые документы в базу;
- подключать внешние научные источники в ответный контур.

## Ключевые фичи

- `Ask workspace`
  Основной сценарий: вопрос, краткий вывод, confidence, факты, доказательства, источники, экспорт.
- `Knowledge graph`
  Граф знаний на Neo4j с query-linked узлами, соседями, публикациями и fallback-режимом.
- `Sources`
  Реестр источников, статусы обработки, количество документов, source typing и управление импортом.
- `Compare`
  Сопоставление решений, фактов, параметров и доказательств между подходами.
- `Gaps`
  Слабое покрытие, недостающие параметры, узкие места корпуса.
- `Curate`
  Экспертная проверка фактов и обработка противоречий.
- `Dashboard`
  Покрытие корпуса, факты, сущности, фрагменты, качество источников и состояние графа.
- `External research module`
  Модуль для поиска релевантных внешних научных материалов и ссылок прямо в ответе.

## Архитектура

Проект собран из пяти сервисов:

- `frontend`
  пользовательский интерфейс на React + Vite
- `api`
  FastAPI control plane, orchestration и публичные endpoints
- `worker`
  фоновая обработка ingestion и extraction jobs
- `postgres`
  операционное хранилище для production-окружения
- `neo4j`
  графовое хранилище для сущностей, связей и graph traversal

Текущий production vertical slice использует:

- полный registry корпуса документов;
- curated deep ingest для demo-критичного поднабора;
- Yandex AI Studio для ответной генерации;
- reverse proxy same-origin схему между фронтендом и API.

## Текущее состояние проекта

Репозиторий подготовлен как стабильный demo-ready vertical slice:

- полный корпус можно загрузить и зарегистрировать;
- curated subset глубоко обрабатывается в фрагменты, факты и graph data;
- UI готов к демонстрации ключевых сценариев;
- production compose уже прогонялся на удаленном Ubuntu-сервере.

Это дает хороший баланс между качеством демо и дальнейшей масштабируемостью.

## Быстрый старт

### Локальный запуск

1. Создайте `.env` на основе `.env.production.example` или задайте переменные окружения вручную.
2. Поднимите стек:

```bash
docker compose up --build
```

3. Откройте:

- Frontend: `http://127.0.0.1:5173`
- API: `http://127.0.0.1:8001`
- Health: `http://127.0.0.1:8001/health`

### Production-like локальный режим

```bash
docker compose -f docker-compose.prod.yml up --build
```

После запуска:

- Frontend: `http://127.0.0.1`
- API health через reverse proxy: `http://127.0.0.1/health`

## Пользовательский demo flow

1. Пользователь открывает workspace.
2. Выбирает сценарий или вводит инженерный вопрос.
3. Получает краткий ответ, confidence и доказательства.
4. Переходит к источникам и фрагментам.
5. Открывает граф знаний по запросу.
6. Сравнивает подходы.
7. Просматривает gaps, contradictions и review state.

## Почему это круто для хакатона

- у проекта есть четкий продуктовый образ, а не просто "дашборд с моделями";
- ответы объяснимы и привязаны к evidence;
- граф знаний — это рабочая часть UX, а не декоративная картинка;
- ingest-подсистема позволяет наращивать корпус, а не жить на одном захардкоженном сценарии;
- архитектура допускает рост в сторону полноценных внешних научных коннекторов;
- решение реально деплоится на одном сервере и не разваливается при показе.

## Что не лежит в Git

В репозиторий намеренно не включены:

- полный корпус научных документов;
- runtime-базы и export-артефакты;
- локальные deployment-архивы;
- `.env` и серверные секреты.

В Git остается сам продукт:

- код приложения;
- manifests и demo data;
- deployment scripts;
- конфигурация UI, API и графового слоя.
