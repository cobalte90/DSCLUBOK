# План реализации проекта "Научный клубок"

R&D Knowledge Graph Copilot для горно-металлургических исследований.

Текущая дата: 2 июля 2026. Дедлайн по постановке: 4 июля 2026 23:59. Значит, стратегия должна быть не "сделать идеальную промышленную платформу", а "сделать убедительный вертикальный срез промышленной платформы": все заявленные классы возможностей должны быть представлены в интерфейсе и backend, а самые сложные сценарии должны работать на реальных документах из корпуса.

## 1. Главная идея

Сделать не просто чат по документам, а систему, которая показывает:

1. Из каких документов извлечен ответ.
2. Какие сущности, параметры, диапазоны и связи попали в граф.
3. Где есть консенсус, где противоречия, где пробелы.
4. Каким источникам можно доверять сильнее.
5. Как эксперт может исправить граф и оставить след аудита.

Рабочее позиционирование для презентации:

> "Научный клубок" распутывает R&D-память института: превращает отчеты, обзоры, статьи, таблицы и презентации в проверяемую карту знаний, по которой можно задавать инженерные вопросы на русском и английском языке.

## 3. Что будет оценено на 10/10

Жюри, скорее всего, будет смотреть не только на наличие RAG, а на соответствие требованиям задачи. Поэтому критерии успеха:

1. Есть импорт разных форматов: PDF, DOCX, XLS/XLSX, PPTX, хотя бы с fallback-режимами.
2. Есть онтология с доменными типами, а не плоские chunks.
3. Есть граф связей: материал -> процесс -> оборудование -> режим -> результат -> источник.
4. Есть поддержка числовых фильтров и единиц измерения.
5. Есть различение географии: Россия, зарубежная практика, мировая практика.
6. Есть мультиязычный словарь терминов: `электроэкстракция`, `electrowinning`, `EW`, `ПВП`, `fluidized bed furnace`.
7. Ответы имеют цитирование источников, уверенность, дату источника и число подтверждений.
8. Есть выявление противоречий и пробелов.
9. Есть ручная правка графа экспертом с аудитом.
10. Есть красивый интерфейс: поиск, фильтры, граф, таблица сравнения, карточки источников, экспорт.
11. Есть понятный deployment через Docker Compose.
12. Есть демонстрационные ответы на 4 вопроса из постановки.

## 4. Продуктовые режимы

### 4.1 Ask

Естественный запрос на русском или английском:

- "Какие методы очистки шахтных вод применимы при сульфатах 200-300 мг/л?"
- "What nickel electrowinning catholyte circulation solutions are described in global practice?"

Вывод:

- краткий ответ;
- таблица методов;
- источники;
- уверенность;
- спорные места;
- подписка на тему или запрос;
- кнопки "показать граф", "экспорт", "создать уведомление".

### 4.2 Explore Graph

Интерактивный граф:

- узлы по типам: Material, Process, Equipment, Parameter, Experiment, Publication, Expert;
- фильтры по году, домену, географии, достоверности;
- подсветка цепочек вида `Материал -> Процесс -> Оборудование -> Режим -> Результат`;
- отдельный цвет для противоречий и пробелов.

### 4.3 Compare

Таблица сравнения технологий:

- эффективность;
- CAPEX/OPEX, если найдено;
- применимость в холодном климате;
- экологические ограничения;
- источники;
- уверенность.

### 4.4 Gaps

Карта пробелов:

- комбинации `материал - процесс - условие`, где мало источников;
- темы, описанные только в отечественных или только в зарубежных источниках;
- параметры с противоречивыми значениями.

### 4.5 Curate

Экспертная правка:

- подтверждать или отклонять извлеченный факт;
- объединять дубли сущностей;
- добавлять комментарий;
- менять уровень доверия;
- видеть историю изменений.

### 4.6 Dashboard

Для руководителя:

- покрытие по доменам: гидрометаллургия, пирометаллургия, экология, отходы;
- количество документов, фактов, экспериментов, источников;
- топ тем с риском противоречий;
- активность экспертов;
- новые документы за период.

## 5. Архитектура

### 5.1 Общая схема

```text
Файлы корпуса
  -> File Registry
  -> Document Converter
  -> Chunker + Table Extractor
  -> Metadata Normalizer
  -> Entity/Relation/Claim Extractor
  -> Unit & Range Extractor
  -> Ontology Validator
  -> Knowledge Graph
  -> Hybrid Search Index
  -> Query Planner
  -> GraphRAG Answer Engine
  -> UI / Export / Audit
```

### 5.2 Рекомендуемый стек для хакатона

Backend:

- Python 3.10+.
- FastAPI.
- Pydantic для контрактов и structured extraction.
- SQLModel или SQLite для пользователей, аудита, задач импорта.
- Neo4j для графа и Cypher-запросов.
- Qdrant или Neo4j vector index для embeddings.
- Elasticsearch/OpenSearch/Vespa как расширение для промышленной версии; в MVP можно начать с BM25 через SQLite FTS или Whoosh, если не успеваем поднять отдельный поисковик.

Frontend:

- React + TypeScript + Vite.
- React Flow или Cytoscape.js для графа.
- TanStack Table для сравнительных таблиц.
- Recharts для дашбордов.

Document processing:

- Для MVP: `python-docx`, `pypdf`, `pandas`, fallback на простое извлечение текста и таблиц.
- Для сильной версии: Docling, потому что он поддерживает PDF, DOCX, PPTX, XLSX, HTML, изображения, OCR, таблицы, Markdown/JSON export и локальный запуск для чувствительных данных.

LLM/NLP:

- Детерминированные правила для чисел, единиц, годов, OIP/ТИ кодов, авторов.
- LLM structured output для сложных сущностей и отношений.
- Доменные словари и синонимы.
- Для локального fallback: ruBERT/Sentence Transformers для embeddings, Natasha/spaCy для NER, rapidfuzz для entity resolution.

Deployment:

- `docker-compose.yml`: `backend`, `frontend`, `neo4j`, `qdrant`, `worker`.
- Один `.env.example`.
- `make ingest-demo`, `make run`, `make test`.

## 6. Онтология

### 6.1 Узлы

| Узел | Назначение |
|---|---|
| `Document` | Файл: путь, формат, размер, hash, уровень доступа. |
| `SourceFragment` | Страница, параграф, таблица или ячейка, откуда взят факт. |
| `Publication` | Статья, обзор, отчет, патент, нормативный документ. |
| `Experiment` | Опыт, испытание, моделирование, промышленная апробация. |
| `Material` | Никель, медь, штейн, шлак, гипс, сульфаты, хлориды. |
| `Process` | Выщелачивание, электроэкстракция, обессоливание, газоочистка. |
| `Equipment` | Ванна EW, диафрагменная ячейка, печь, фильтр, скруббер. |
| `Facility` | Рудник, фабрика, цех, установка, полигон, лабораторная площадка. |
| `Property` | Химическое, физическое, технологическое или экономическое свойство. |
| `Parameter` | Температура, концентрация, pH, скорость потока, плотность тока. |
| `Condition` | Холодный климат, география, режим, тип сырья. |
| `Result` | Выход, извлечение, качество продукта, расход, экономический эффект. |
| `Claim` | Вывод или рекомендация, которые можно подтвердить или опровергнуть. |
| `Expert` | Автор, исполнитель, носитель компетенции. |
| `Organization` | Лаборатория, институт, предприятие. |
| `Geo` | Страна, регион, предприятие, "Россия", "зарубежная практика". |
| `Topic` | Тематический тег. |
| `AccessPolicy` | Метка доступа и чувствительности. |
| `ChangeSet` | Версия ручной или автоматической правки. |

| Связь | Пример |

### 6.3 Свойства факта

Каждый факт должен иметь:

- `fact_id`;

- `value`, если факт числовой;
- `unit`;
- `min_value`, `max_value`;
- `condition`;
- `geo`;
- `time_period`;
- `source_fragment_id`;
- `confidence`;
- `verification_status`: `extracted`, `verified`, `rejected`, `needs_review`;
- `updated_at`;
- `access_level`.

Приоритет 2:

- доклады и свежие журналы 2024-2025.

Приоритет 3:

- архивы и старые рыночные отчеты.

### 7.2 Document Converter

MVP:

- DOCX: `python-docx`.
- PDF: `pypdf`, page-level extraction.
- XLS/XLSX: `pandas` или fallback metadata-only.
- PPTX: `python-pptx`, если успеваем.
- DOC/RAR/ZIP: metadata-only, плюс "unsupported yet" в UI.

10/10 stretch:

- Docling как единый конвертер с OCR, таблицами, reading order и JSON/Markdown export.
- Отдельный статус документа: `parsed`, `partial`, `failed`, `metadata_only`.

### 7.3 Chunking

Чанки должны сохранять структуру:

- документ;
- страница;
- заголовок раздела;
- номер таблицы;
- параграф;
- текст;
- соседние чанки;
- embedding;
- language.

Размер:

- 700-1200 токенов для русского текста;
- overlap 100-150 токенов;
- таблицы не дробить построчно без необходимости, а хранить как отдельные `TableFragment`.

### 7.4 Извлечение чисел и единиц

Это нельзя отдавать только LLM. Нужен детерминированный слой:

1. Regex для диапазонов: `200-300 мг/л`, `60-65 °С`, `<=1000 мг/дм3`.
2. Нормализация единиц: `мг/л`, `мг/дм3`, `г/л`, `%`, `°C`, `А/м2`, `т/сут`.
3. Приведение десятичных запятых.
4. Связь числа с ближайшим параметром: `сульфат`, `температура`, `плотность тока`, `pH`.
5. Проверка адекватности по доменным диапазонам.

Пример структуры:

```json
{
  "parameter": "temperature",
  "material": "электролит",
  "min_value": 60,
  "max_value": 65,
  "unit": "degC",
  "source_fragment_id": "doc_12_p_8_para_3",
  "confidence": 0.91
}
```

### 7.5 LLM structured extraction

Для каждого чанка:

```json
{
  "entities": [
    {
      "type": "Process",
      "name": "электроэкстракция никеля",
      "aliases": ["nickel electrowinning", "Ni EW"]
    }
  ],
  "relations": [
    {
      "source": "электроэкстракция никеля",
      "type": "OPERATES_AT",
      "target": "60-65 °С"
    }
  ],
  "claims": [
    {
      "text": "Сульфатный электролиз никеля обычно ведут при 60-65 °С",
      "polarity": "positive",
      "confidence": 0.82
    }
  ]
}
```

Важно: LLM не является источником факта. Источник факта - документ и фрагмент. LLM только извлекает структуру.

### 7.6 Entity resolution

Объединять:

- `электроэкстракция`, `электролитическое извлечение`, `electrowinning`, `EW`;
- `ПВП`, `печь взвешенной плавки`, `flash smelting furnace`;
- `МПГ`, `платиновые металлы`, `PGM`, `platinum group metals`;
- `шахтные воды`, `mine water`, `MIW`.

Алгоритм:

1. Словарь синонимов.
2. Fuzzy match по нормализованному имени.
3. Embedding similarity.
4. Проверка типа сущности.
5. Если confidence ниже порога, отправить в очередь curator review.

## 8. Модель доверия и верификации

Формула уверенности:

```text
confidence =
  0.30 * source_quality
+ 0.25 * extraction_quality
+ 0.20 * corroboration
+ 0.15 * recency
+ 0.10 * curator_status
```

Где:

- `source_quality`: тип источника, внутренний обзор, статья, патент, журнал, презентация;
- `extraction_quality`: найден ли фрагмент, число, единица, таблица, связь;
- `corroboration`: сколько независимых источников подтверждают факт;
- `recency`: актуальность;
- `curator_status`: подтверждено экспертом или нет.

Статусы:

- `extracted`: факт извлечен автоматически;
- `needs_review`: низкая уверенность или конфликт;
- `verified`: подтвержден экспертом;
- `rejected`: отклонен;
- `superseded`: заменен более новым источником.

Противоречия:

- одинаковый параметр, материал и процесс, но диапазоны не пересекаются;
- противоположные выводы в claims;
- один источник говорит "применимо", другой "не применимо" для тех же условий.

## 9. Поиск и ответы

### 9.1 Query Planner

Запрос пользователя превращается в структуру:

```json
{
  "intent": "technology_review",
  "materials": ["никель"],
  "processes": ["электроэкстракция"],
  "parameters": [
    {"name": "скорость потока", "operator": "optimal"}
  ],
  "geo": ["world"],
  "time_range": null,
  "need_comparison": true,
  "need_graph": true
}
```

Типы intent:

- `fact_lookup`;
- `technology_review`;
- `compare`;
- `find_experiments`;
- `find_experts`;
- `knowledge_gap`;
- `contradiction_check`;
- `export_report`.

### 9.2 Retrieval

Использовать гибридный поиск:

1. Lexical BM25/FTS для точных терминов, формул, аббревиатур.
2. Vector search для смысловой близости.
3. Graph traversal для связей 2-4 уровней.
4. Numeric filters через свойства графа.
5. География и год через metadata filters.
6. Reranking с учетом источника, свежести, совпадения параметров.

Почему это важно:

- чистый vector search часто плохо работает по числам, единицам и аббревиатурам;
- чистый keyword search не найдет синонимы;
- граф нужен, когда ответ требует связать материал, процесс, оборудование и результат.

### 9.3 Answer Engine

Шаблон ответа:

1. Краткий вывод.
2. Таблица вариантов или фактов.
3. Условия применимости.
4. Числовые параметры.

## 17. Риски и решения

| Риск | Что делать |
|---|---|
| 5.2 ГБ не успеем полноценно распарсить | Полный registry + deep indexing для приоритетных документов. |
| PDF бывают сканами | В MVP помечать `needs OCR`, в stretch подключить Docling/OCR. |
| LLM ошибается с числами | Числа извлекать regex/parser, LLM только связывает их с сущностями. |
| Граф получится слишком шумным | Schema-guided extraction + confidence threshold + curator queue. |
| Демо зависит от долгой индексации | Сделать `demo` snapshot и precomputed graph. |
| Сложно поднять много сервисов | MVP запускается на FastAPI + Neo4j + frontend; Qdrant/Elastic как optional profile. |
| Нет API ключей | Сделать mock answer mode и возможность подключить локальные embeddings. |

## 18. "Больше, чем просили"

### 18.1 Топ-10 value features

Это 10 фич, которые сильнее всего поднимают практическую ценность проекта и должны считаться приоритетными после закрытия базовых требований.

1. Evidence View: у каждого ответа есть режим "почему система так решила" с цепочкой `факт -> фрагмент -> документ -> confidence -> дата актуализации`.
2. Contradiction Resolver: система не скрывает конфликтующие данные, а показывает различия по источникам, годам, диапазонам и дает эксперту возможность пометить "актуальнее", "спорно" или "нужно проверить".
3. Scenario Builder: пользователь задает материал, процесс, ограничения, климат, CAPEX/OPEX и получает сравнительную таблицу технологий под конкретный инженерный сценарий.
4. Research Gap Map: матрица `материал x процесс x условие`, которая показывает плохо изученные комбинации и зоны, где не хватает данных для решения.
5. Expert Finder: поиск экспертов, лабораторий и команд, уже работавших с похожими материалами, процессами или режимами.
6. Table-First Extraction: приоритетное извлечение таблиц, потому что ключевые параметры, режимы и технико-экономические показатели часто находятся именно там.
7. Experiment Passport: карточка эксперимента с сырьем, режимом, параметрами, результатом, выводом и привязкой к источнику.
8. Saved Queries + Alerts: сохраненные запросы и подписки на тему с уведомлением о новых документах, фактах или конфликтах.
9. Report Generator: сборка готового мини-обзора в `Markdown/PDF/JSON-LD` для презентации, ТЗ, служебной записки или отчета.
10. Source Quality Ranking: ранжирование источников по типу и надежности: внутренний отчет, обзор, статья, патент, презентация, справочник.

### 18.2 Почему именно они

- Эти фичи напрямую помогают принимать инженерные решения, а не просто искать текст.
- Они хорошо видны в демо и легко объясняются жюри через реальные кейсы корпуса.
- Они создают разницу между обычным RAG по документам и настоящей R&D knowledge platform.

## 22. Ресерч и почему такие решения

Использованные источники:

- Microsoft GraphRAG: https://microsoft.github.io/graphrag/  
  Важное: GraphRAG строит knowledge graph из текста, извлекает entities/relationships/claims, делает community summaries и поддерживает Global/Local/DRIFT search. Это ровно подходит под запросы, где нужно "связать точки" в большом частном корпусе.

- Neo4j GraphRAG Python: https://neo4j.com/docs/neo4j-graphrag-python/current/  
  Важное: официальный пакет Neo4j поддерживает RAG, vector indexes, knowledge graph builder pipeline, schema-guided extraction, entity resolver. Это хороший аргумент для Neo4j как центрального графового слоя.

- Neo4j KG Builder guide: https://neo4j.com/docs/neo4j-graphrag-python/current/user_guide_kg_builder.html  
  Важное: pipeline состоит из loader, splitter, embedder, schema builder, entity/relation extractor, KG writer, entity resolver. Эту же схему можно адаптировать под наши документы.

- Docling: https://github.com/docling-project/docling  
  Важное: поддерживает множество форматов, включая PDF, DOCX, PPTX, XLSX, изображения, OCR, таблицы, Markdown/HTML/JSON export и локальное выполнение для чувствительных данных.

- W3C SHACL: https://www.w3.org/TR/shacl/  
  Важное: стандарт для валидации RDF-графов через shapes. Полезен для промышленной версии: проверять, что факт имеет источник, единицу, confidence и provenance.

- GO FAIR principles: https://www.go-fair.org/fair-principles/  
  Важное: findable, accessible, interoperable, reusable. Для презентации можно показать, что система не просто отвечает, а делает R&D знания FAIR-compatible.

- Vespa hybrid search tutorial: https://docs.vespa.ai/en/learn/tutorials/hybrid-search.html  
  Важное: гибрид lexical + vector search, BM25, embeddings, ranking profiles, RRF/normalization. Это подтверждает выбранную архитектуру retrieval.

- Elasticsearch kNN docs: https://www.elastic.co/docs/solutions/search/vector/knn  
  Важное: kNN, filtered vector search, hybrid retrieval с query + knn. Полезно как промышленная альтернатива Vespa/OpenSearch.

## 23. Самое важное для победы

Не пытаться продать "мы загрузили PDF в чат". Нужно демонстрировать:

1. У нас есть граф, он виден и редактируем.
2. У каждого вывода есть источник и уверенность.
3. Числа и единицы работают как структурированные данные.
4. Запросы проходят через материал + процесс + условия + географию + время.
5. Система честно говорит "нет данных" или "есть противоречие".
6. На реальных документах из корпуса уже есть полезные ответы.

Короткий слоган для финала:

> "Научный клубок превращает архив R&D в проверяемую инженерную память: от документа к факту, от факта к графу, от графа к решению."
